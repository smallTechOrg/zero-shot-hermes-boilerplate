# Zero-Shot Hermes Harness — project context

This repo is a **working multi-agent hackathon boilerplate** (Hermes-native). The runnable code
lives directly in the repo — there is no generator step. It is the project context Hermes loads
whenever the working directory is inside this repo (`.hermes.md` is discovered by walking parents
up to the git root).

## What this is

A lean, opinionated starter-kit for building agentic software **spec-first**. One person with an
idea and one API key can drive a real, production-shaped agent into existence in 20–60 minutes.
The stack is conventional and reviewable: FastAPI + React/Vite + SQLite, with a Hermes goal loop.

The convictions:

1. **Spec is the source of truth.** Written before code; when spec and code disagree, the spec wins.
2. **Built for two audiences at once.** A non-coder drives it with a sentence; a senior engineer inherits a clean, reviewable stack.
3. **Lean boilerplate, not a framework.** `harness/` is engineering mindfulness — kept small.
4. **Smallest first-time-right win.** Each slice must work the *first* time it is tested.
5. **A human gates each milestone.** Autonomous within a slice; test before moving on.
6. **Real LLM/API or it doesn't count.** Evals run against the real model with keys from `.env`.

## How Hermes uses this repo

- **Entry point:** `AGENTS.md` (root) — read first every session.
- **Rules:** `harness/rules/ai-agents.md`, `harness/rules/git.md`, `harness/rules/secret-hygiene.md`.
- **Patterns:** `harness/patterns/*` (spec-driven, phases, project-layout, tech-stack, code, test-driven, ui-ux, agentic-ai, engineering-practices, goal-loop).
- **Skills (stubs):** `harness/skills/<name>/SKILL.md` — read + follow; optional to install as native Hermes skills.
- **Sub-agent roles:** `harness/agents/*.md` — invoked via Hermes `delegate_task`.
- **Spec:** `spec/` — the product spec (roadmap, architecture, capabilities, data, api, ui, agent).
- **Actual code:** `backend/app/` (FastAPI agent + routers), `frontend/` (React chat UI), `deploy/` (GCP), `capabilities/`, `evals/`.

## Layout (actual)

```
backend/app/   FastAPI: main.py, agent.py (supervisor/worker + Gemini), config.py, db.py, models.py, routers/
frontend/      React + Vite + Tailwind chat UI
deploy/        GCP VM: Dockerfile, gcp-vm-startup.sh, terraform/main.tf, cloudbuild.yaml
capabilities/  one-file-per-capability plugin convention
harness/       Hermes skill/agent/pattern stubs
spec/          product spec (source of truth)
evals/         pytest behaviour + live evals
```

## Run it

```bash
cd backend && python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt && uvicorn app.main:app --port 8000
# frontend:
cd frontend && npm install && npm run dev
# evals:
cd backend && .venv/bin/python -m pytest ../evals/ -v
```

Go live: copy `.env.example` → `.env`, set `GEMINI_API_KEY`, restart the backend. `POST /api/chat`
then calls Gemini instead of returning `[stub]`.

## Hard rules (survive context compression)

- README must always be accurate — every command works exactly as written.
- Never claim a test passed if you didn't run it. Show output.
- App code lives on a feature branch (`feature/v0.1`); its PR targets the branch it was cut from, **never `main`** unless explicitly intended as the boilerplate release.
- Commit + push are one atomic action: `git commit … && git push origin <branch>`.
- Stage specific files. `.env` is never staged.
- Real-key testing only. A stubbed pass is not a pass.
